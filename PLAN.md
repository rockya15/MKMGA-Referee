# MKMGA Referee — System Plan

## Overview

A Jackbox-style game system for **MKMGA** — a game where players watch a streamer play Mario Kart and bet on where he places.

The host computer is connected to the TV. It runs the full host application (displayed on the TV), the local game server, and a Python sidecar that watches the YouTube stream for race results. Players join from their phones via a public URL generated by Cloudflare Tunnel.

## Architecture

### Host App (React — fullscreen browser on TV)
- Displays embedded YouTube video via YouTube IFrame API
- Host pastes YouTube URL once; video plays + sidecar starts watching simultaneously
- Game UI overlays float as a HUD over the video (Jackbox-style)
- Two views:
  - `/host` — the TV display (fullscreen, no controls)
  - `/host-controls` — a second tab/device the host uses to drive the game
- **Idle Mode:** If the host hasn't interacted with the program for 30+ seconds (no tab focus, no mouse/keyboard activity):
  - The TV display automatically enters "Leaderboard Loop" mode
  - Cycles through each remaining player, displaying their **focused card** with stats for 5 seconds per player
  - Stats shown: name, balance, bets won, money spent, preferred position, random profile question answer
  - Loop continues until the host returns and clicks/moves to exit idle mode
  - Provides ambient entertainment during downtime between races

### Local Server (Node.js + Express + Socket.io)
- Runs entirely on the host computer
- Manages all game state: `LOBBY → PRE_BET → BETTING → PAYOUT → GAME_OVER`
- Trusts events from `/host-controls` and the Python sidecar (shared secret token)
- Broadcasts state updates to all connected player phones
- **Image Upload & Storage:**
  - Serves `/api/upload-profile` endpoint for profile image uploads during join
  - Stores images locally in `/uploads/player-images/` directory
  - Images are temporary and deleted when the game ends
  - Max file size: 5MB; allowed formats: JPEG, PNG, WebP

### Python Sidecar
- Receives the YouTube URL from the server
- `yt-dlp` pulls the stream directly from YouTube servers (independent of the embedded player)
- `ffmpeg` extracts a frame every ~2 seconds
- Detection logic identifies the Mario Kart results screen and reads the placement
- Emits `RACE_COMPLETE` with placement data to the local server via Socket.io

### Cloudflare Tunnel
- Punches a stable public HTTPS URL from the local server
- Players use this URL to join from their phones — no port forwarding needed
- Launched automatically as part of the startup script

### Startup
One batch script (or Docker Compose) launches everything:
1. Node.js server
2. Python sidecar
3. Cloudflare Tunnel

## Player Mobile Experience

### Focus vs. Idle States
Player phones operate in two modes at all times:

**Idle** — the player is not being asked to do anything right now
- A **leaderboard** is available and can be opened at any time
- The leaderboard is split into two sections:
  - **Sorted players** — players whose positions have been confirmed for this race
  - **Unsorted players** — players still waiting to be assigned a position (e.g. mid-wheel-spin during Stage 2)
- Each entry shows the player's display name, standing, and current balance
- During the wheel/position assignment phase, this updates live so players can follow along

**Active** — the player is being asked to make a decision (Pay/Skip, position pick, bet action, vote)
- Any open popup or overlay (e.g. leaderboard) is **immediately closed**
- The phone goes full-focus on the current prompt with large, clear action buttons
- No other navigation is accessible until the action is completed or timed out

## Game Stages

### Stage 0 — Host Configuration (before lobby opens)
Before the join URL becomes live, the host sets the following on the host-controls panel:
- **Maximum cash cap** — the highest amount any player is allowed to bring (must be a multiple of $0.25)
- The lobby does not open and the join URL does not accept players until the host confirms these settings and presses **Open Lobby**

---

### Stage 1 — Lobby (Setup)

#### Joining
- Players open the public URL on their phone
- If the game has already started, they are shown a **"Game in progress — you cannot join mid-game"** screen and denied entry
- If the lobby is open, they are taken to the join form

#### Join Form — Step 1: Identity
- **Display name** (username) — can be anything, even funny/fake names
- **Real name / nickname** — what the host and system will use to identify them
- No two players may share the same display name or real name. If a duplicate is detected, the second entry is **auto-denied** with an explanation
- Host can **clear any existing account** from the host-controls panel to resolve edge cases (e.g. someone typed a joke username that conflicts with a real player)

#### Join Form — Step 2: Cash
- Player enters how much cash they brought
- **Maximum cash cap** applies (set by host in Stage 0 before the lobby opened)
- Minimum is any amount expressible in quarters (must be a multiple of $0.25)
- Player selects their denomination mix:
  - "I brought all quarters"
  - "I brought dollar coins (and quarters)"
- **Profile Image Upload (Optional):**
  - Player may select an image file from their device
  - Image is previewed before submission
  - File is uploaded to the local server during form completion
  - Image will be displayed on their player card during the game
- A **warning is shown** during onboarding: *"You are responsible for keeping track of your own coins during the game. Failure to do so may result in disqualification."*

#### Join Form — Step 3: Profile Questions
- **5 Random Personality Questions:**
  - Player is shown 5 randomly-selected questions from a larger pool (e.g., "What's your lucky number?", "First video game you played?", "Favorite Mario Kart track?", etc.)
  - Player provides short text answers to each question
  - **Visibility Warning:** The player is made explicitly aware that "*Other players will be able to see your answers to these questions.*"
- One answer is randomly selected and stored to display on the player's card during focused moments
- Answers are saved with the player's profile and persist throughout the game session

#### Player Cards & Leaderboard
- **Card Design:**
  - **Default state (Vertical):** Shows profile image (1:1 square, centered), display name below
  - **Focused state (Horizontal):** Card expands horizontally, displays profile image, display name, and **statistics:**
    - Current balance
    - Total bets won this session
    - Total money spent this session
    - Preferred position (most frequently selected position across races)
    - Randomly-selected profile question answer (displayed as flavor text)
- If a player uploaded a profile image, it is displayed as a square on their card
- Leaderboard during play shows sorted players (position confirmed) and unsorted players (awaiting assignment)
- Images and answers persist throughout the session and appear on the TV display and other players' phones

#### Rejoin
- If a player's phone crashes or they close the browser, they can reopen the URL and enter their real name again to be restored to their session
- Their balance, position, game state, and profile image are preserved on the server
- If they reconnect while their turn is still being held (within the 30s window), they are dropped straight back into their active prompt
- If they reconnect after the group vote has already resolved their action, nothing is being asked of them — they land on the idle page with the leaderboard available

#### Elimination Screen
- When a player is knocked out of the game (balance reaches $0), their phone displays a full-screen **skull and crossbones** screen
- No further interaction is possible from that device
- **Lucky Bastard Respawn Process:**
  - After a player is eliminated, the Host may determine (via an external game or process entirely unrelated to MKMGA — e.g., blackjack, a race, coin flip, etc.) whether that player is a **Lucky Bastard**
  - If a player is chosen as a Lucky Bastard, they are respawned with their **base credits** (verified by the host at registration to prevent fraud)
  - **Respawn Timing:** The respawn is **immediate upon death** — it must occur before the player would pick their next position. The player cannot hold a respawn and use it later in the game.
  - **Final 4 Constraint:** Once the game reaches the final 4 players (competition is fierce), no more Lucky Bastards may be chosen
  - The Host manually controls the respawn via the `/host-controls` panel

#### Mid-Game Join Denial
- Any device that opens the URL after the game has started receives a denial screen
- No exceptions — players cannot join mid-game

#### Host Controls
- Host sees a live-updating player list on `/host-controls`
- Host presses **Start Game** → server transitions to `PRE_BET`

### Stage 2 — Pre-Bet
- Host tabs to YouTube to show players which map the streamer is playing
- Host returns to the app and advances to Pre-Bet
- TV shows the current **Entry Fee** prominently
- Each player's phone shows two buttons: **Pay Entry Fee** / **Skip**
- Players who pay proceed to the Wheel Spin, then Stage 3 (Betting)
- Players who skip sit out this race (their balance is unchanged)

#### The Wheel
After all Pay/Skip decisions are collected, the TV displays a **spinning wheel** containing all paying players' names.

- The wheel spins and lands on a player at random
- That player immediately **picks their position(s)** (see Position Assignment below)
- The wheel spins again — the landed player is removed, remaining players are on the wheel
- This repeats until all paying players have been assigned positions
- **Spin order = betting order for Stage 3.** "Previous players" during raises means players who were spun earlier.

#### Position Assignment
There are **13 positions**: placements **1–12** and **DNF** (streamer rage-quits).

**More than 13 paying players:**
- Free choice — players may pick any position, including ones already chosen by others
- Multiple players can share a position and split the winnings if it hits
- DNF is a freely available pick like any other position. There is no cascade. There is no drama. If you want DNF, you may have DNF.

**13 or fewer paying players — Position Distribution:**
When 13 or fewer players pay, all 13 positions must be claimed. Positions are distributed per this formula:
> `base = floor(13 ÷ n)` — minimum picks per player  
> `remainder = 13 mod n` — how many players get one extra pick  
> The first `remainder` spinners (earliest spun) get `base + 1` picks. Everyone else gets `base`.

| Players | Base picks | Who gets +1 |
|---------|-----------|-------------|
| 12 | 1 | 1st spinner |
| 11 | 1 | 1st & 2nd spinner |
| 7 | 1 | 1st–6th spinner |
| 6 | 2 | 1st spinner |
| 5 | 2 | 1st–3rd spinner |
| 4 | 3 | 1st spinner |
| 3 | 4 | 1st spinner |
| 2 | 6 | 1st spinner |
| 1 | 13 | — |

When a player has multiple picks, they choose all of them immediately when the wheel lands on them — before it spins again.

**The Cascade Rule:**

When a player is assigned DNF (either as their sole remaining pick or as one of many picks), they have two options:

1. **Accept DNF** — keep it, no further action
2. **Trigger the cascade** — invoke the DNF cascade wheel

**Cascade Spending Rules:**
- Only **one cascade chain per position assignment phase** can ever be triggered
- A cascade chain is considered complete (spent) when it ends via any of: a player lands on a free position, a player lands on DNF, or a player is forced to accept DNF by the wheel
- Once a chain completes, **no further cascades may be started** in that round — any subsequent DNF assignments must be accepted

**Cascade Trigger Consequences:**

The cascade wheel's starting probability depends on ***whether the current player was forced to take DNF or chose it voluntarily:***

**Forced DNF (Gentle Cascade Table):** The player had zero other options available. All other positions are taken or unavailable. This player is forced into DNF. They may accept it or cascade. They start at **level 1** of the gentle table.

**Chosen DNF (Harsh Start, ~31% base):** The player had other available positions but deliberately selected DNF. This is a gamble — either accept DNF, or cascade. If they cascade, they're betting on the wheel to displace another player into DNF, creating chaos. **The punishment:** they begin at ~31% probability.

If this early cascader succeeds and displaces another player into DNF, that displaced player enters the **gentle cascade table at the next level** — not restarting harsh. This localizes the penalty to the scummy early-picker.

**Gentle Cascade Table (Forced DNF or Downstream Displaced):**

| Level | DNF slots | Position slots | DNF chance |
|-------|-----------|----------------|------------|
| 1 | 1 | 12 | ~8% |
| 2 | 2 | 11 | ~15% |
| 3 | 4 | 9 | ~31% |
| 4 | 8 | 5 | ~62% |
| 5 | 13 | 0 | 100% |

**Harsh Cascade Start (Chosen DNF Only):**
Used **only by the player who initiated the cascade** by choosing DNF when alternatives existed.

| Spin | DNF slots | Position slots | DNF chance |
|------|-----------|----------------|------------|
| 1st | 4 | 9 | ~31% |
| 2nd | 8 | 5 | ~62% |
| 3rd | 13 | 0 | 100% |

**Strategic Implication:**
There is only one cascade per round. If you select DNF early and cascade, you start at harsh ~31%. If you succeed and displace another player, that player enters the gentle cascade at level 2 (~15%), continuing from where you left off. This incentivizes players to either:
- Take their preferred position and leave DNF for the last player (who starts at gentle level 1, ~8%), or
- Accept DNF immediately if forced (enters gentle level 1), preserving cascade for another player

**Within a Cascade Chain:**
- Landing on a free position ends the chain — the cascading player takes it, equilibrium restored
- Landing on a taken position displaces that player to DNF (they may accept it or, if the cascade chain has not yet ended, trigger it themselves at the next gentle level)
- Landing on DNF ends the chain unconditionally — the cascading player must accept
- At gentle level 5 or harsh spin 3, DNF is guaranteed — the chain always ends
- If a player using the harsh cascade is displaced back to DNF by a downstream player reclaiming their position, they must continue their harsh cascade progression, not switch to gentle logic. This means they face increasing DNF probabilities on subsequent spins, potentially leading to guaranteed DNF.

### Stage 3 — Betting

Poker-style betting in wheel spin order.

#### Actions
| Action | Description |
|--------|-------------|
| **Check** | Pass — no additional bet (only if no raise has occurred) |
| **Raise** | Increase the current bet |
| **Call** | Match the current raised bet |
| **Fold** | Drop out of this race's pot |

#### Rules
- Turn order follows **wheel spin order** (first spun = first to act)
- Players may raise multiple times per round
- When a player raises, all players who acted **before** the raiser are re-queued — they must **Call** or **Fold** only. Re-queued players may not raise again. Only players who have not yet acted in the current raise cycle may raise.
- **Bet cap = the lowest-funded paying player's remaining balance at the start of the betting phase** (after entry fee, before any bets). Non-payers excluded. Once the cumulative bet hits that ceiling:
  - The Raise option is removed from all remaining players' UIs
  - Players who haven't yet matched the current bet can only **Call** or **Fold**
  - Players who have already matched can only **Check** or **Fold**
  - Raising stops entirely; betting continues only as Calls, Checks, or Folds
- Players who fold forfeit all money already put into the pot — no refunds
- Players who cannot meet a raise must go **All In** or **Fold** (if they still have their token) — otherwise they must go **All In**
  - A player going All In that completes a call does not get re-queued; their action is finalized
- **If a group vote on behalf of a timed-out player results in a Fold, that Fold consumes the timed-out player's Skip/Fold token**
- Raise amounts are coin-friendly increments: **$0.25, $0.50, $1.00, $2.00** (or custom input)
- **The cap is recalculated only if a player folds.** All In does not recalculate the cap upward. Example: Player 1 (floor at $2, $2 remaining) folds after Player 2 raises to $3. Player 3 can now raise higher than Player 2 could, but Player 2 (who already matched at $3) is closed out and cannot re-enter even though cap changed.

#### Timer
- Each player has **60 seconds** to act
- A GSAP-animated timer bar displays on both the TV and the player's phone
- Timer turns red in the final 15 seconds
- **On timeout:** a 30-second **group vote** opens for all other active players
  - Each voter chooses what the timed-out player should do (available actions depend on game state: Check/Fold, or Call/Fold)
  - Majority vote wins; ties broken by the host
  - The voted action is executed on the timed-out player's behalf
  - The timed-out player's phone shows a "waiting on vote" state during this window

#### Visual Effects (TV display, driven by Socket.io events)
- **Spotlight** — active player is highlighted with a radial gradient sweep (GSAP)
- **Pot raise burst** — giant dollar sign animates in + particle explosion on every raise (GSAP + canvas-confetti)
- **Timer bar** — red urgency shift in final 15 seconds
- Effect architecture is fully event-driven — new effects can be added by listening to new Socket.io events, no game logic changes needed

### Stage 4 — Payout
- Python sidecar automatically detects the results screen and reads the streamer's placement
- Server receives `RACE_COMPLETE` event with placement
- Server identifies winner(s) (players who bet on the matching placement)
- **If a player is eliminated mid-race** (edge-case elimination after already paying the entry fee and being assigned a position), their position is treated as vacated. If the streamer finishes in that position, no active player holds it — the pot carries over to the next race per the standard no-winner rule.
- Pot is split **equally** among winners, rounded down to the nearest **$0.25** (quarters are the minimum denomination — no cents, no dimes)
- Any remainder that can't be cleanly split rolls into a **seed pot** that is added to the next race's pot
- TV shows animated payout: money flying to winner cards, updated leaderboard
- Updated balances are reflected on all player phones
- **Lucky Bastard Check:** If any players were eliminated this race, the Host checks the `/host-controls` panel for respawn opportunities
  - Host may play an external game (unrelated to MKMGA) with each eliminated player to determine if they're a "Lucky Bastard"
  - If a player is chosen for respawn and the game has more than 4 players remaining, the Host clicks `respawn` on the `/host-controls` panel for that player
  - Respawned players are restored with their base credits and rejoin the game
  - Their phone is reactivated and they proceed to the next race's position assignment
- **Host Pauses:** The server pauses and awaits host action on the respawn panel. Once respawn decisions are finalized (or the host confirms no respawns), the Host presses **Next Race** → race counter increments, loops back to Stage 2

---

### Game Over
- Triggered when only one player has a remaining balance
- Any remaining funds in the pot are awarded to the last player standing
- TV shows: winner crown animation, full session summary
  - Per-player earnings/losses
  - Number of races played
  - Final standings
- **Cleanup:** All player profile images stored in `/uploads/player-images/` are automatically deleted when the game terminates
- **The King of MKMGA** is the last player standing — since all players must eventually go bankrupt, the winner is whoever outlasts everyone else. If the game ends early (e.g. the group stops playing before anyone is eliminated), the King is whoever has the highest total winnings at that point. If participants are too intoxicated and someone passes out, the last player still conscious is the King. If a tie exists (same earnings, same consciousness level), The Host and the tied participants work out a plan outside of the game itself that everyone agrees on to settle who is King.
  - **King Prizes & Legacy:** The First King, Konspired, earned a monkey. The Second King, Lizzie, earned a trophy of the racer we all bet on. The next King will earn something incredible.

## House Rules

These are enforced by the group, not the software:

- The pot may not be touched. Punishment: thrown into the pool. Always. No exceptions.
- Ads = round of shots / everyone sips.
- Each player has exactly **one Skip/Fold token** per night, shared between skipping a race and folding during betting. Once spent, they must Pay every race and can only Call or go All In.
- A group vote that results in a Fold on behalf of a timed-out player consumes that player's token.
- Players who fold do not get their money back.
- If you cannot meet a raise and have no token, you must go All In.
- If no one wins (or the winning position is vacated by an eliminated player), the money stays in the pot and carries to the next race
  - **Special case: All-at-Zero Continuity** — If all paying players are left at exactly $0.00 due to All In matching the entry fee (common at Race 10+), the next race uses **the exact same position assignments** as the current race. Players do not re-spin positions; they retain what they had. This allows the pot to be won despite no one having remaining balance. If the same positions hit again, the pot is won. If not, it continues to cascade forward.
- The last player standing is declared **The King of MKMGA**. If the game ends early, whoever has the highest total winnings becomes King. If someone passes out from intoxication, the last conscious player is King. If a tie exists, The Host and tied players work out a plan outside the game to settle it.
- **Lucky Bastard Rules** (determined by external game, at Host discretion):
  - A respawned Lucky Bastard is restored to their original buy-in amount
  - Respawn is mandatory-immediate: if chosen, must occur before the next race's position assignment
  - Only one Lucky Bastard respawn per player, per game
  - Once the game reaches final 4 players, no more respawns are permitted
  - The Host controls all respawn decisions via the host-controls panel

## Detection: How the Sidecar Works

```
YouTube URL (pasted by host)
        │
        ▼
    yt-dlp taps stream directly from YouTube servers
        │
        ▼
    ffmpeg extracts one frame every ~2 seconds
        │
        ▼
    OpenCV template matching
    (compare frame to reference screenshot of MK results screen)
        │
      match?
       / \
     Yes   No → discard frame, wait for next
      │
      ▼
    (optional) Vision LLM reads placement from frame
        │
        ▼
    Emit RACE_COMPLETE { placement: N } to local server via Socket.io
```

The embedded YouTube player and yt-dlp pull from YouTube independently — the embed is for display only, yt-dlp bypasses the browser entirely.

## Decisions Log

1. **Raise rules** — When a player raises, all players who acted before them are re-queued but may only Call or Fold — not raise again. Only players who haven't acted in the current raise cycle may raise.
2. **Bet cap** — Bet cap = lowest paying player's remaining balance at the start of the betting phase (after entry fee, before bets). Non-payers excluded. Cap **only recalculates if a player folds**. All In does not recalculate the cap upward; a player completing a call via All In is finalized and doesn't get re-queued.
3. **Timer expiry** — 30-second group vote by all other active players; majority wins, host breaks ties. If the vote result is Fold, it consumes the timed-out player's Skip/Fold token.
4. **Disconnection** — Turn is held for 30s to allow reconnect, then group vote triggers if still gone.
5. **Placement betting** — Players bet on the streamer's exact finishing placement (1–12 or DNF).
6. **Split pot denomination** — All payouts round down to the nearest $0.25. Remainder seeds the next race's pot.
7. **Skip/Fold token** — Each player has one token per night shared between Skip and Fold. No token = must Pay and must Call or All In.
8. **Mid-race edge-case elimination** — If a player is eliminated after paying the entry fee, their position is vacated. If the streamer hits that position, the pot carries.
9. **Zero payers** — Race skips entirely; entry fee does not escalate; pot carries.
10. **Solo payer / Race 10+** — Betting phase is skipped; race proceeds directly to payout.
11. **Cascade spending** — Only one cascade chain per position assignment phase (applies when 13 or fewer players pay). Cascade chain is spent when it ends via: free position, DNF, or forced acceptance. Initiator of early cascade gets harsh ~31% wheel as punishment; any displaced players downstream continue at gentle table next level (~15%, ~31%, ~62%, ~100%).
12. **All-at-Zero continuity** — If all paying players reach exactly $0.00 via All In (common at Race 10+), the next race repeats the same position assignments instead of respinning. Allows pot to be won despite zero balances.
13. **King determination** — Last player standing in normal flow. In early-end: highest earnings. If someone passes out: last conscious player. If tie: Host + tied players work out a plan outside the game that all agree on.
14. **Player admin controls** — Host can kick players, set balances, and set positions live from host controls. Server is authoritative and emits full game-state updates after each action.
15. **Session safety events** — Kicked players and full-game resets force player clients back to the main menu state.
16. **Host view composition** — TV layout uses modular active-element + leaderboard + footer segmentation, with active panel hidden when no active element exists.
17. **Active element transition gating** — Active panel content only begins after entry transition settles; exit handoff delays leaderboard expansion for cleaner slide-out.
18. **Votes shown in active panel** — Group vote and position vote displays render in active-element area rather than footer.
19. **Wheel segment color source** — Position-assignment wheel segment colors derive from each player's favorite color (validated hex with fallback).
